1) Path traversal при сохранении raw (POST /api/raw_save)
Проблема
- Имя файла от клиента использовалось напрямую (RAW_DIR / name), что позволяло записать файл за пределами RAW_DIR, если name содержал "../".

Возможные решения
a) safe_join + валидация имени (уже реализовано)
   - Использовать безопасную операцию объединения пути (resolve) и проверку, что результат находится внутри RAW_DIR.
   - Плюсы: простое и надёжное решение для предотвращения path traversal.
   - Минусы: относительные пути и симлинки могут вести к неожиданностям; нужно учитывать нормализацию.

b) Ограничить набор допустимых имён (whitelist)
   - Разрешать только имена, соответствующие регулярному выражению, например: ^[\w\-\u0400-\u04FF\s]+\.txt$
   - Плюсы: предотвращает все необычные имена, простая валидация.
   - Минусы: менее удобна для пользователя (ограничивает символы), потребуется отказаться от некоторых международных символов либо расширить правило.

c) Игнорировать имя клиента — генерировать имя на сервере
   - Сервер принимает содержимое и генерирует уникальное имя (например, timestamp + hash + оригинальное расширение), сохраняет маппинг user_visible_name -> stored_name.
   - Плюсы: полностью исключает возможность path traversal и коллизий; удобно для многопользовательской среды.
   - Минусы: изменяет UX (пользователь не видит тот же путь), требует хранения маппинга.

Рекомендация
- Для простоты и обратной совместимости: сохранить safe_join и добавить дополнительную проверку регуляркой (вариант a + b). Для мульти‑пользовательских сценариев — вариант c.


2) Небезопасный filename в Content-Disposition (/api/file_download)
Проблема
- Content-Disposition включал имя напрямую, что может привести к нечёткой интерпретации заголовка при наличии кавычек или контрольных символов.

Возможные решения
a) Базовая санитизация (basename + удаление проблемных символов)
   - Уже реализовано: os.path.basename + удаление двойных кавычек и контролей.
   - Плюсы: простое исправление, совместимо со многими клиентами.
   - Минусы: не покрывает все случаи (не экранирует non-ASCII по RFC), но работает в большинстве браузеров.

b) Использовать RFC 6266 способ (filename* с кодированием UTF-8)
   - Формировать заголовок: Content-Disposition: attachment; filename="fallback.txt"; filename*=UTF-8''%D1%82%D0%B5%D1%81%D1%82.txt
   - Плюсы: корректная передача UTF-8 имён в соответствии со стандартом.
   - Минусы: чуть более сложная реализация; некоторые старые клиенты могут не поддерживать filename*.

Рекомендация
- Совместить оба подхода: sanitize + добавить filename* для UTF-8. Для MVP достаточно базовой санитизации.


3) Наивный парсинг CSV и возможный XSS во frontend (index.html)
Проблема
- index.html парсит CSV с помощью split(',') и вставляет HTML через innerHTML. Это ломается, если поля в кавычках содержат запятые, и позволяет XSS при наличии HTML в данных.

Возможные решения
a) Серверный парсинг CSV в JSON
   - Реализовать /api/file_json?book=&name=..., который читает CSV на сервере с помощью csv.reader и отдаёт JSON: {headers: [...], rows: [[...], ...]}.
   - Плюсы: простота, безопасная передача структурированных данных; фронтенду не нужно парсить CSV.
   - Минусы: увеличивает нагрузку на сервер при больших файлах; нужно учитывать память/стриминг.

b) Клиентский CSV‑парсер (PapaParse)
   - Включить в index.html PapaParse (или другой лёгкий парсер) и парсить CSV корректно, затем вставлять ячейки как текст (textContent).
   - Плюсы: не требует серверных изменений, корректный парсинг, поддержка больших файлов (стриминг) у PapaParse.
   - Минусы: добавляет внешнюю библиотеку в SPA; немного больше работы на фронтенде.

c) Экранировать ячейки и улучшить split
   - Временно: экранировать каждый cell (создавать text node) и использовать более умный split, но это ненадёжно для сложных CSV.
   - Плюсы: быстро исправляет XSS; не требует больших изменений.
   - Минусы: CSV парсинг всё ещё ненадёжен.

Рекомендация
- Реализовать серверный /api/file_json для CSV меньшего размера (<= X MB). Для более крупных файлов использовать PapaParse с textContent. Либо сначала добавить серверный endpoint — проще и безопаснее.


4) Блокирующий синхронный запуск анализа (subprocess.run без очереди)
Проблема
- /api/run_analysis запускает анализ синхронно и ждёт завершения в thread handler. Долгие запуски могут блокировать потоки, приводить к тайм‑аутам и плохому UX.

Возможные решения
a) Фоновый запуск с простым файловым job queue
   - При запросе /api/run_analysis создать job dir (jobs/<id>) и положить туда JSON с параметрами; запустить worker (отдельный thread/process) или cron, который берёт jobs в фоне и выполняет анализ, записывает stdout/stderr и помечает статус.
   - Предоставить /api/job_status?id=... и /api/jobs.
   - Плюсы: простая реализация, надёжна, легко логируется.
   - Минусы: нужно реализовать менеджер очереди и синхронизацию.

b) Использовать threading.Thread для запуска subprocess в фоне и возвращать immediately
   - Запускать thread при получении запроса, thread выполняет subprocess.run и записывает результаты в outputs; возвращать job_id на старте и опрашивать статус.
   - Плюсы: быстрая реализация, минимум инфраструктуры.
   - Минусы: при перезапуске сервера background threads теряются; требует хранение состояния на диске для устойчивости.

c) Перейти на полноценный task queue (RQ/Celery) — heavy
   - Использовать Redis + RQ/Celery для управления задачами и мониторинга.
   - Плюсы: промышленное решение для фоновых задач.
   - Минусы: требует дополнительных сервисов и сложнее в Termux/локальной среде.

Рекомендация
- Для MVP: вариант (b) или (a) — реализовать простую очередь с filesystem-backed jobs и background worker thread, минимально (правило: jobs survive server restart).


5) Отсутствие логирования и сохранения stdout/stderr процесса
Проблема
- stdout/stderr анализов обрезаются и не сохраняются в outputs для последующего дебага.

Возможные решения
a) Всегда сохранять полный stdout/stderr в outputs/<book>/run.log и отдавать урезанный фрагмент в API
   - Плюсы: отладка и пост‑мортем возможны, нет потери информации.
   - Минусы: большие логи требуют дискового пространства, нужно очищать старые логи.

b) Собирать только последние N байт, но дополнительно сохранять полный вывод в ротационный лог
   - Плюсы: компромисс между доступностью информации и контролем диска.

Рекомендация
- Сохранять полный run.log, но вводить ротацию/лимит по размеру (например, keep last 5 logs, max 5MB each).


6) safe_join edge cases и симлинки
Проблема
- resolve() может следовать симлинкам и обойти проверки; также unicode/normalize проблемы могут вызывать bypass.

Решения
- Привести base и target к realpath и проверить, что общий путь имеет base как префикс; ещё можно запрещать симлинки внутри RAW_DIR или разрешать только при явной конфигурации.

Рекомендация
- Оставить safe_join, но логировать подозрительные resolve() различия и, если нужна повышенная безопасность, отвергать симлинки в RAW_DIR.


7) Другие мелкие улучшения и быстрые фикс‑ideas
- Добавить Content-Type корректный при отдаче CSV (text/csv; charset=utf-8) и при скачивании: set Content-Length и safe filename.
- Добавить заголовок Cache-Control: no-store для динамических endpoint'ов, если нужно.
- Улучшить frontend UX: показывать прогресс/статус анализа, показать ссылки «скачать» рядом с превью, превью первых N строк + кнопка «скачать весь файл».
- Защитить server-bind: по умолчанию слушать 127.0.0.1; при запуске на 0.0.0.0 предупредить в логах.
