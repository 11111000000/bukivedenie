# Master plan: MVP CLI/API для извлечения статистики из текста (Termux, без LLM)

Цель документа
- зафиксировать единый концентрат технического задания и контракт для MVP, который реализует детерминированный конвейер извлечения данных из художественных/публицистических текстов и построение интерактивных визуализаций. MVP без LLM; опционально возможность включать heavier модули при флагах, но по умолчанию — отключены.
- обеспечить выполненность на платформе Termux (Android) с Python 3.10+; минимальный набор зависимостей через requirements.txt; heavy‑модели (spaCy large, transformers, sbert) — по флагу с предупреждением.

1) Область применения и границы
- Вход: один файл .txt (одна книга) или директория с .txt файлов (каждый файл — своя книга). Поддержка UTF‑8.
- Языковая поддержка: ru, en, auto; смешанные тексты допускаются;
- Типы текстов: художественная проза, диалоги, публицистика; в рамках MVP — не используем NER/embeddings/LLM; работа с базовыми эвристиками.
- Архитектура: CLI‑скрипт (analyze_text.py) + библиотечный API (extractor.extract_stats) с REST‑контрактом по необходимости; экспорты на диск; возможность запуска в локальном/CLI режимах; heavy‑функции можно отключать через флаги.

2) Входные данные и конфигурация
- Форматы входа: plain .txt; возможна поддержка .jsonl по требованию.
- Кодировка: UTF‑8.
- Детекция глав: авто‑детекция по regex; возможность переопределить через --chapter-pattern (пример: r'^(Глава|CHAPTER|Chapter\s*\d+)')
- Опции конфигурации: язык, chapter-pattern, режимы анализа и объема (см. раздел CLI).

3) Архитектура и компоненты (MVP)
- Backend: Python 3.10+, Typer‑CLI и/или FastAPI для API (если нужен REST‑слой); детерминированный конвейер анализа текста; хранение результатов в памяти с экспортом на диск.
- Frontend: веб‑UI (React/Vue/Svelte) с библиотекой графиков (ECharts/Chart.js) или статическими HTML‑страницами — для MVP можно отказаться и полагаться на экспортированные файлы и простой просмотр.
- API контракт: извлечение статистики возвращает view‑ready данные; экспорт файлов реализуется через outputs/ деревья.
- Режимы исполнения: локальный CLI, локальный сервер; опционально — headless batch режим.

4) Конвейер извлечения данных (детерминированный)
- Текст и offsets
  - raw_text — исходник; canonical_text — нормализация; Offsets по raw_text (0-based, end exclusive).
  - offset_map — будущий модуль сопоставления canonical↔raw (позже).
- Нормализация
  - NFC; унификация кавычек/многоточий/тире; CRLF→LF; нормализация пробелов; удаление управляющих символов.
- Абзацы
  - Разделение на абзацы: 2+ переводов строки; poetry_mode — 1 перенос; is_dialog — строка начинается с тире/кавычек.
- Предложения
  - Эвристика разбиения: границы по [.!?…] с учётом кавычек/скобок; исключение аббревиатур/инициалов/дат/URL; флаг is_dialog/contains_quote.
- Токенизация
  - Unicode‑aware: token types: word, number, punct, symbol, mixed, abbrev, roman_numeral, url, email.
  - word_count включает word (и roman_numeral при наличии).
- Структурные единицы
  - Секции/главы: эвристики (напр., «Глава», «Часть», заглавные строки) и эпиграфы.
- Сегменты/окна
  - Окна сегментов: базовая сетка по предложениям (например, win_sent=50, step=25); relative_pos в [0,1].
- Диалог/цитаты
  - Детекция прямой речи: диалоговые предложения помечаются как is_dialog; кавычки/тире — сигнальные признаки.
- Кандидаты имён (NER, rule‑based)
  - heuristic: последовательности заглавных слов; учёт составных имён; частоты и first_offset.
  - при включении NLP‑моделей (опционально): spaCy/Natasha – но по умолчанию выключено.
- Совместная встречаемость (ко‑occurrence)
  - ребра между двумя именами внутри одного сегмента (sentence/paragraph) с весом = число совместных появлений.

5) Выходные данные и визуализация (view-ready)
- outputs (папка outputs/<book_id>/):
  - tokens.csv (или parquet) — частоты токенов
  - chapters_summary.json — свод по главам
  - characters.csv — список обнаруженных персонажей + частоты
  - character_freq_by_chapter.csv
  - cooccurrence_edges.csv — edges: source,target,weight,context_level
  - embeddings/ — embeddings (npy или tsv) по токенам/фрагментам/главам
  - sentiment_by_chapter.csv
  - frequency_dictionary.csv — token,count,rank,per_1k
  - hapax.csv
  - complexity_metrics.json — Yule's K, Honore's R и пр.
  - run_metadata.json — runtime, flags, версии моделей и правил
- форматы: CSV/JSON по умолчанию; Parquet — опция (--parquet)

6) CLI и API
- CLI: analyze_text.py (Typer) и модуль extractor.py (бэкенд‑логика)
- Обязательные параметры:
  - --input <path> (файл или директория)
  - --output-dir <dir>
  - --lang <ru|en> (или auto)
  - --chapter-pattern <regex> (опционально)
- Опции/флаги:
  - --ner [off|heuristic|spacy|natasha] (default heuristic)
  - --embeddings [off|fasttext|sbert] (default off)
  - --sentiment [off|lexicon|vader|transformer] (default lexicon)
  - --cooccurrence-level [sentence|paragraph]
  - --use-pos (включение POS‑пометки для плотности лексики)
  - --parquet (вывод Parquet)
  - --workers N
  - --verbose, --dry-run
- API контракт
  - extractor.extract_stats(text: str, book_id: str, out_dir: Path, config: dict) -> dict
  - CLI вызывает extractor.extract_stats, возвращает summary.

7) Алгоритмы и эвристики (ключевые точки)
- Главы: авто‑детекция, override через --chapter-pattern
- Токены и сегменты: razdel для ru; nltk/regex fallback для en; offsets; token_lower
- Диалоги: правила на основе тире/кавычек; вычисление доли реплик
- NER/персонажи: heuristic по заглавным словам; опционально spaCy/Natasha
- Ко‑occurrence: по предложению/по абзацу
- Embeddings: optional, легковесные варианты (fastText) и heavy‑модели (SBERT) — при необходимости
- Тональность/плотность лексики: lexicon-based sentiment; словари; опционально transformer‑модель
- Лексическая сложность: Yule’s K, Honore’s R, hapax legomena, лексическая плотность

8) Архитектура и требования к ресурсам (Termux)
- Платформа: Android Termux; Python 3.10+; pinned‑версии в requirements.txt
- Ограничения: без крупных ML‑моделей; heavy‑функции — опционально через флаги
- Память/CPU: разумное использование, параллельность через --workers; кеширование
- Логи/версионирование: run_metadata.json, версия правил и моделей

9) Тестирование и релизы
- Бazовые unit‑тесты на глав, токены, детекцию диалогов, NER‑ fallback; интеграционные тесты на end‑to‑end для одной книги.
- Документация по установке в Termux и примеры использования CLI.

10) Этапы внедрения
- Этап 1: MVP CLI + базовая аналитика (главы, персонажи, частоты, ко‑окружение, частотный словарь, hapax, сложность) + экспорт CSV/JSON
- Этап 2: вставка NER/embeddings/тональности/лексической сложности; опциональные зависимости
- Этап 3: экспорт Parquet, улучшенная визуализация + REST API (при необходимости)
- Этап 4: CI/тесты, документация

11) Контроль критических факторов
- Унификация контракта выходов; offsets; включение heavy‑моделей только по флагу; тестовое покрытие; четкая документация
