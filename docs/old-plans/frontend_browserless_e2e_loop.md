# План: browserless end-to-end цикл для фронта и визуализаций

## Цель
Сделать минимальный цикл разработки, где изменения во фронтенде и визуализациях можно проверять без ручного открытия браузера на каждом шаге.

## Что уже есть
- `frontend/rollup.config.js` запускает `rollup -w` с `serve` и `livereload`.
- `frontend/src/api.js` уже направляет локальный фронт на `http://127.0.0.1:8000`.
- `src/webapp.py` отдает API и статику.
- Есть ручной smoke-план: `tests/frontend_smoke.md`.

## Минимальный целевой цикл
1. Запустить backend на `127.0.0.1:8000`.
2. Запустить Rollup watch для `frontend/`.
3. Запустить headless smoke-harness.
4. При падении исправить первое сообщение об ошибке.
5. Повторить цикл.

## Что должен проверять smoke-harness
- `/api/books` отвечает и возвращает список.
- `/api/files?book=...` отвечает для фикстуры.
- `/api/file_parsed?book=...&name=...` возвращает ожидаемый тип данных.
- Критические маршруты фронта рендерятся без JS-ошибок:
  - `#/books`
  - `#/book/:id`
  - `#/book/:id/viz/tokens`
  - `#/book/:id/viz/wordcloud`
  - `#/book/:id/viz/network`
  - `#/book/:id/viz/sentiment`
  - `#/book/:id/viz/heatmap`
  - `#/book/:id/files`
  - `#/book/:id/file/:name`
- Для каждой визуализации проверять хотя бы наличие mount-элемента и отсутствие console errors / failed requests.

## Тезис, антитезис, синтез
- Тезис: текущий Rollup + backend уже достаточен для локальной разработки.
- Антитезис: без headless-проверки мы видим только факт загрузки страницы, но не факт успешного рендера визуализаций.
- Синтез: добавить один небольшой headless smoke-скрипт и запускать его вместе с dev-циклом.

## Почему именно минимально
- Не вводить новый фреймворк.
- Не строить собственный UI-devtool.
- Не добавлять image diff как обязательный первый шаг.
- Не ломать существующую схему `frontend/` + backend.

## Предлагаемая реализация
### Фаза 1. Канонический smoke-командный вход
- Добавить одну команду вида `make ui-smoke` или `npm run smoke`.
- Команда поднимает или использует уже запущенный backend и фронт.

### Фаза 2. Headless browser harness
- Использовать Playwright или Puppeteer.
- Открывать `http://127.0.0.1:5173`.
- Переходить по маршрутам.
- Валидировать `console.error`, network failures и наличие chart containers.

### Фаза 3. Маленький API smoke до браузера
- Проверять базовые API ответы до запуска browser-step.
- Это позволит ловить поломки данных раньше, чем падение рендера.

### Фаза 4. Документация
- Зафиксировать рабочий цикл в `README.md`.
- Оставить `tests/frontend_smoke.md` как fallback для ручной проверки.

## Диалектическая критика
- Сильная сторона подхода: минимальная сложность, быстрый feedback loop, меньше ручной работы.
- Слабое место: headless smoke ловит структурные ошибки, но не тонкие визуальные регрессии.
- Компромисс: начать с DOM/console/assertion smoke, а screenshot diff добавить только если реально появится боль.
- Риск: отдельные порты и CORS могут скрыть интеграционные проблемы, если smoke не проверяет реальные маршруты и реальные API ответы.

## Definition of Done
- Один запуск поднимает dev-окружение и smoke.
- Ошибки API и рендера видны без ручного клика по UI.
- Для каждого ключевого маршрута есть стабильная проверка.
- Документирован рабочий цикл для дальнейшей доработки визуализаций.
