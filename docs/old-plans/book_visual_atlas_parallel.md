# План: визуальный atlas книг через параллельных субагентов

## Цель
Собрать фронтенд так, чтобы книга воспринималась как красивый интерактивный atlas, а не как набор отдельных страниц и таблиц.

## Диалектический разбор
### Тезис
Уже есть рабочий dashboard shell, selected-book контекст, route fallback и несколько источников данных из текущего backend/API.

### Антитезис
Сейчас большая часть виджетов ещё выглядит как превью или таблица, а не как полноценная визуализация книги. Нужны сильные графики, единый atlas-центр и более выразительный визуальный язык.

### Синтез
Не переписывать интерфейс. Сохранить dashboard shell, но заменить ключевые панели на настоящие визуализации и один общий atlas-контур.

## Бритва Оккама
Самый быстрый и надёжный путь:
1. не делать новый frontend framework;
2. не создавать новый роутинг;
3. не переносить всю аналитику в backend;
4. добавить только те библиотеки и визуальные компоненты, которые дают заметный прирост качества.

## Визуальный MVP
Первый релиз должен включать:
- Book Atlas как центральную панель;
- Top Words как полноценный chart;
- Punctuation Timeline как компактную визуализацию ритма текста;
- Text Viewer с контекстом выбранной книги/файла;
- Files summary как навигационный блок;
- минимум один graph/network view с хорошей компоновкой.

## Рекомендуемый стек
- `echarts` для большинства графиков и компактных dashboard views.
- `echarts-wordcloud` для word cloud.
- `cytoscape` для графов/сетей.
- `vega-lite` оставить только там, где он уже дешевле, чем переписывать.

## Параллельные субагенты
### Субагент A: Atlas shell и layout
**Задача:** превратить `#/books` в визуальный atlas-экран.

**Что сделать:**
- усилить центральную роль `Book Atlas`;
- добавить визуальную иерархию между atlas и вспомогательными панелями;
- сделать более выразительные карточки и плотность компоновки;
- сохранить route-based fallback.

**Файлы:**
- `frontend/src/views/booksList.js`
- `frontend/src/style.css`
- возможно, `frontend/src/router.js`

### Субагент B: Чартовые виджеты
**Задача:** заменить текстовые превью на реальные графики там, где это уже возможно.

**Что сделать:**
- Top Words -> bar/tile chart;
- Punctuation Timeline -> timeline/heatmap-like chart;
- при необходимости вынести общие chart helpers.

**Файлы:**
- `frontend/src/views/tokensChart.js`
- `frontend/src/views/bookOverview.js`
- `frontend/src/views/booksList.js`
- `frontend/src/viz/*` при необходимости

### Субагент C: Network / word cloud / strong visuals
**Задача:** довести самые “красивые” панели до сильного визуального уровня.

**Что сделать:**
- word cloud на `echarts-wordcloud` или оставить текущий вариант, если он уже достаточен;
- network view на `cytoscape`-style компоновку либо улучшить текущий вид;
- сделать визуальные ограничения по высоте и скроллу.

**Файлы:**
- `frontend/src/views/wordCloud.js`
- `frontend/src/views/networkGraph.js`
- `frontend/src/style.css`

### Субагент D: Smoke и контракты
**Задача:** не сломать dashboard-first flow при визуальном усилении.

**Что сделать:**
- обновить smoke contract, если появляются новые ready markers;
- проверить route ordering;
- сохранить artifact naming и browser fallback;
- зафиксировать новые визуальные зависимости в тестах.

**Файлы:**
- `scripts/ui_smoke.mjs`
- `frontend/tests/ui_smoke.test.mjs`
- `tests/frontend_smoke.md`

### Субагент E: Dependencies / package.json
**Задача:** добавить нужные библиотеки без избыточности.

**Что сделать:**
- обновить `package.json`;
- добавить только те библиотеки, которые реально дают качество визуализации;
- сохранить install/build/test/smoke стабильными.

**Файлы:**
- `frontend/package.json`
- `frontend/package-lock.json`

## Порядок выполнения
1. Зафиксировать atlas shell и компоновку.
2. Подключить/улучшить chart libraries.
3. Довести network/word cloud до визуального уровня.
4. Проверить smoke и артефакты.
5. Прогнать build/test/smoke.

## Критерии готовности
- dashboard ощущается как atlas книги;
- ключевые виджеты выглядят как графики, а не как заглушки;
- selected-book контекст понятен;
- route fallback продолжает работать;
- smoke и build остаются зелёными.

## Принцип остановки
Если виджет не улучшает чтение книги или не даёт заметной визуальной ценности, он откладывается во вторую фазу.
