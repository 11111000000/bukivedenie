# План: dashboard-first развитие через параллельных субагентов

## Контекст
Текущий фронтенд уже умеет показывать отдельные маршруты и визуализации, но для реальной разработки этого мало. Нужен один основной экран-дэшборд, где:
- список книг всегда актуален и берётся из данных на диске;
- у книги виден статус готовности данных;
- можно запустить майнинг/анализ, если данных ещё нет;
- выбор книги обновляет все виджеты сразу;
- каждый виджет компактный, с собственным скроллом при необходимости, с заголовком и кратким описанием.

## Диалектический анализ
### Тезис
Есть рабочая база: backend уже отдаёт API, фронт уже умеет рендерить книги, файлы и отдельные визуализации, а также есть smoke-цикл и локальный dev-loop.

### Антитезис
Существующая схема слишком фрагментирована: отдельные страницы, локальные допущения по данным, недостаток статуса готовности, нет единого dashboard-экранa, нет общего обновления по выбранной книге.

### Синтез
Не переписывать систему. Нужен минимальный слой над уже существующими маршрутами и API: один дашборд, один источник правды для списка книг и статуса, один понятный цикл запуска данных, один набор компактных виджетов.

## Бритва Оккама
Надёжнее и быстрее всего не строить новый фреймворк, не плодить новые abstraction layers и не переносить всю логику в один большой backend endpoint. Самый простой рабочий путь:
1. оставить существующие визуализации;
2. собрать над ними dashboard shell;
3. добавить только те backend-данные, которых не хватает для списка книг, статуса и запуска анализа;
4. проверить всё smoke-скриптом.

## Цель
Сделать dashboard-first интерфейс, который показывает актуальные данные по выбранной книге и поддерживает быстрый цикл разработки без ручного обхода множества экранов.

## Не-цели
- Не переписывать фронтенд на React/Vue/Svelte.
- Не вводить новый визуальный движок.
- Не делать image-diff как обязательный первый шаг.
- Не ломать существующие route-based экраны, если они уже полезны как fallback.

## Что должно получиться
- Главный экран — дашборд, а не набор отдельных страниц.
- Слева или сверху — список книг.
- При выборе книги дэшборд обновляет все виджеты.
- Виджеты имеют фиксированную, компактную высоту.
- Если виджет перегружен данными, он скроллится внутри себя.
- У каждого виджета есть:
  - название;
  - короткое описание;
  - основная визуализация или таблица;
  - состояние загрузки или ошибки.
- Если данные не смайнены, виден явный CTA для запуска пайплайна.

## Главный образ
Дашборд должен восприниматься не как набор карточек, а как интерактивный атлас книги.

- Центральный виджет: `Book Atlas` или `Карта книги`.
- Остальные панели работают как увеличительные стекла к нему.
- Клик по любому графику должен вести назад к фрагменту текста.
- Текстовый просмотрщик обязателен, он связывает аналитику и чтение.

## Путь Лао Цзы
Чтобы ускорить работу и не потерять качество, идём от простого к полезному.

- Один источник правды для данных.
- Один главный экран.
- Один текстовый индекс.
- Один связанный текстовый просмотрщик.
- Три базовые визуализации в первой фазе: карта книги, топ слов, пунктуация по ходу текста.
- Всё остальное только после того, как этот контур стабилен и быстрый.

Правило: если визуализация не помогает чтению книги уже сейчас, она уходит в фазу 2.

## Визуализации дашборда
Один экран, без вкладок. Вместо навигации по разделам — сетка компактных карточек, которые синхронно обновляются по выбранной книге.

- Book Atlas: линейная карта книги по ходу текста, с главами, абзацами, репликами, пунктуационными всплесками и аномалиями.
- Архитектура текста: sunburst / icicle, длины глав, гистограмма абзацев.
- Лексика: топ-N слов, кривая покрытия, облако слов.
- Формы и длины: длина слова, длина предложения.
- Пунктуация: профиль знаков, теплокарта по ходу текста, circle packing.
- Диалоги: реплики по главам, длина реплик.
- Дополнительные графы: сеть персонажей или связей, если она нужна для конкретного корпуса.
- Дополнительные акценты: нити мотивов, ритм фраз как кардиограмма, аномалии и переломы.

## Фазы
### Фаза 1: быстрый и качественный MVP
- `Book Atlas`
- топ слов
- пунктуация по ходу текста
- text viewer с подсветкой
- минимальный compare между двумя фрагментами

### Фаза 2: расширение
- мотивы
- диалоги
- сеть персонажей
- coverage curve
- anomaly views

## Рекомендуемый стек визуализации
- ECharts как основной declarative engine для баров, линий, heatmap, sunburst, treemap и compact dashboard charts.
- Cytoscape.js для графов и network views.
- `echarts-wordcloud` для облака слов.
- Vega-Lite оставить как вспомогательный слой только там, где он реально короче; не тащить его в основной путь.
- Без React/Vue/Svelte и без лишнего abstraction layer.

## Контракт взаимодействия
- Hover показывает точные абсолюты.
- Click фиксирует выбранный объект и подсвечивает его во всех виджетах.
- Shift-click добавляет к сравнению.
- Double click уводит в текстовый фрагмент.
- Brush по оси текста фильтрует все панели.
- Escape снимает выбор.

## Что ещё нужно от backend
Чтобы поддержать больше визуализаций, backend должен отдавать не только готовые CSV/JSON, но и единый слой индексированных текстовых данных.

- `book_index` или `text_index`: книга -> части -> главы -> абзацы -> предложения -> токены.
- `fragments` с `start/end` offsets для карты книги и brush-синхронизации.
- `chapter_stats` для компактных карточек и fingerprints.
- `punctuation_timeline` для heatmap и партитуры пунктуации.
- `token_coverage` / `vocab_coverage` для кривой покрытия.
- `motif_series` для фазы 2.
- `dialogue_spans` и `dialogue_speakers` для фазы 2.
- `anomalies` для фазы 2.
- `compare` payload для сравнения двух книг или двух фрагментов одной книги.

Это не обязательно делать одним гигантским endpoint. Лучше собрать 2-4 небольших контрактных endpoint'а и кешировать их рядом с анализом.

## Общий state
Нужен один событийный слой без фреймворков.

- `selectedBook`
- `selectedChapter`
- `selectedFragment`
- `selectedToken`
- `selectedCharacter`
- `selectedMotif`
- `selectedPunctuation`
- `activeBrushRange`

## Идеи backend-эндпоинтов
Минимальный набор, который даст рост визуализаций без переписывания всего API:

- `/api/book_index?book=` - дерево книги и offsets.
- `/api/book_summary?book=` - компактные абсолютные метрики и fingerprints.
- `/api/book_fragments?book=&size=` - последовательные фрагменты для atlas и heatmap.
- `/api/motif_series?book=&term=` - абсолютное появление мотива по фрагментам.
- `/api/dialogues?book=` - реплики, длины, спикеры, если доступны.
- `/api/punctuation_timeline?book=&size=` - знаки по ходу текста.
- `/api/compare_books?a=&b=` - данные для сравнительного режима.

Если endpoint тяжёлый, он должен строить cached artifact в `outputs/` и потом только отдавать JSON.

## Индекс текста
Чтобы графики были связаны с чтением, дашборду нужен единый индекс разметки.

- `token_idx`
- `chapter_id`
- `paragraph_id`
- `sentence_id`
- `fragment_id`
- `char_start`
- `char_end`

## Производительность
- Тяжёлые виджеты грузить лениво.
- Дорогие раскладки выносить в worker.
- Для плотных матриц и длинных лент держать canvas-first рендер.
- Не пытаться рисовать всё сразу при открытии книги.
- Не строить то, что нельзя быстро показать.

## Backend performance
- Кешировать промежуточные JSON/CSV артефакты после анализа.
- Не пересчитывать токены и разметку на каждый hover.
- Для atlas и heatmap отдавать уже собранные массивы, а не сырой текст.
- Для сравнений использовать отдельные precomputed payloads.

## Минимальная архитектура решения
### Backend
Нужен не новый backend, а минимальное расширение текущего:
- список книг должен идти из актуальной папки данных;
- для каждой книги нужен статус: готова/не готова/ошибка;
- нужен запуск анализа или майнинга через явную кнопку;
- дэшборд должен получать агрегированный payload с данными, которые нужны для всех виджетов.
- нужны отдельные payload'ы для atlas, motifs, dialogues, punctuation timeline, coverage и compare mode;
- нужно явное кеширование тяжёлых метрик.
- сначала собрать быстрые payload'ы для atlas, top words и punctuation timeline; остальные выдавать позже.

### Frontend
Нужен один dashboard shell:
- селектор книги;
- карточки-виджеты;
- единый loading/error state;
- переиспользование существующих renderers для tokens, wordcloud, network, sentiment, heatmap, files;
- центральный `Book Atlas` как основной вход в книгу;
- визуализации строятся как declarative specs, а не как набор отдельных страниц и вкладок.

### Smoke / verification
Нужен один канонический smoke-запуск:
- API preflight;
- загрузка dashboard;
- переключение книги;
- базовые route checks для fallback-страниц;
- сбор логов и артефактов.

## План для параллельных субагентов
### Субагент A: Backend и источник правды
**Задача:** сделать список книг и статус данных актуальными.

**Что проверить:**
- откуда реально берётся список книг;
- какие файлы/папки считаются готовыми данными;
- какие endpoints уже существуют для запуска анализа;
- нужен ли один агрегирующий endpoint для dashboard payload.

**Файлы, которые вероятно затронет:**
- `src/webapp.py`
- возможно, `tests/` для API contract checks

**Результат:**
- актуальный список книг;
- явный статус готовности;
- кнопка/endpoint запуска анализа;
- минимальные тесты на контракт.

### Субагент B: Dashboard UI и виджеты
**Задача:** собрать аккуратный dashboard-first экран.

**Что сделать:**
- добавить dashboard layout;
- разместить список книг;
- показать summary и статус;
- встроить виджеты в карточки;
- сделать фиксированную высоту и внутренний scroll там, где это нужно;
- добавить заголовок и краткое описание каждому виджету.

**Файлы, которые вероятно затронет:**
- `frontend/src/main.js`
- `frontend/src/router.js`
- `frontend/src/views/*.js`
- `frontend/src/ui/*` при необходимости
- `frontend/src/styles/*` если есть или аналогичные CSS-файлы

**Результат:**
- один главный экран дашборда;
- визуально аккуратная сетка;
- выбранная книга обновляет всё сразу;
- старые страницы остаются fallback-ом.

### Субагент C: Smoke, artifacts и быстрый цикл
**Задача:** закрыть цикл проверки и не дать регрессиям пройти незамеченными.

**Что сделать:**
- расширить smoke-процедуру под dashboard;
- проверить API preflight;
- проверить переключение книги;
- собрать console/log/html/screenshot артефакты;
- добавить короткую документацию по запуску.

**Файлы, которые вероятно затронет:**
- `scripts/ui_smoke.mjs`
- `frontend/tests/*`
- `README.md`
- `docs/plans/*` если нужно уточнение контракта

**Результат:**
- один понятный запуск для контейнера и Termux-сценариев;
- воспроизводимые артефакты;
- быстрое обнаружение поломок.

## Порядок выполнения
### Этап 1. Снять поверхность и зафиксировать контракт
1. Подтвердить, где лежит источник списка книг.
2. Подтвердить, какие данные реально нужны для dashboard payload.
3. Зафиксировать минимальный набор виджетов.
4. Уточнить, какие данные обязаны быть precomputed, а какие можно считать по кнопке.

### Этап 2. Развести параллельную работу
1. Backend-субагент делает источник правды и API.
2. UI-субагент собирает dashboard shell и карточки виджетов.
3. Smoke-субагент готовит проверки и артефакты.

### Этап 3. Свести всё в один поток
1. Прогнать build.
2. Прогнать smoke.
3. Исправить первое реальное падение.
4. Повторить до стабильности.

## Приоритеты по MVP
1. Актуальный список книг.
2. Статус данных и кнопка запуска пайплайна.
3. Dashboard layout.
4. Book Atlas и связанный text viewer.
5. Общий state и синхронизация виджетов.
6. Обновление всех виджетов по выбранной книге.
7. Аккуратная вёрстка карточек.
8. Smoke и артефакты.
9. Backend payloads для atlas, motifs, dialogues и compare.
10. Пунктуация по ходу текста как второй обязательный слой.

## Критерии готовности
- Список книг не устаревает.
- Выбор книги обновляет dashboard без ручной навигации по множеству экранов.
- Виджеты компактны и читаемы.
- Главный экран ощущается как atlas книги, а не как набор разрозненных графиков.
- Клик по любому ключевому виджету ведёт к связанному текстовому фрагменту.
- Если данных нет, пользователь видит понятный путь к запуску анализа.
- Smoke подтверждает, что API и UI живы.
- Backend отдает стабильные JSON-представления для новых визуализаций без повторного сканирования текста на каждом запросе.
- MVP должен запускаться быстро даже на больших книгах.

## Риски
- Если слишком рано свернуть всё в один большой endpoint, станет хуже поддерживать.
- Если оставить отдельные экраны без dashboard shell, UX останется фрагментированным.
- Если не проверить актуальность списка книг из папки, интерфейс будет показывать устаревшие данные.
- Если не зафиксировать smoke, регрессии будут возвращаться.

## Итог
Самый быстрый надёжный путь — не новый стек, а тонкий dashboard layer над существующим кодом, с параллельной работой backend/UI/smoke и строгим ограничением на объём изменений.

## Задачи по файлам
### 1. Backend / источник правды
**Цель:** актуальный список книг и статус данных из папки.

1. Определить, какая папка является источником списка книг.
2. Добавить/уточнить backend-функцию, которая читает книги прямо из файловой системы.
3. Добавить статус готовности данных для каждой книги.
4. Добавить явный endpoint или существующий вызов для запуска анализа/майнинга.
5. При необходимости добавить агрегированный dashboard endpoint, чтобы фронт получал всё одним запросом.
6. Добавить тесты на контракт API.
7. Добавить payload для `Book Atlas` и текстовых offsets.
8. Добавить payload для мотивов, диалогов и пунктуационной ленты.
9. Добавить compare-endpoint для двух книг/двух фрагментов.
10. Встроить кеширование результатов анализа и промежуточных артефактов.
11. Отдельно зафиксировать минимальные payload'ы первой фазы.

**Вероятные файлы:**
- `src/webapp.py`
- `tests/test_webapp_endpoints.py` или новый API test

### 2. Dashboard UI

2. Добавить список книг в левую панель или верхний селектор.
3. Добавить переключение книги без ухода с dashboard.

7. Ограничить высоту виджетов и включить внутренний scroll там, где нужно.
8. Показать loading / empty / error states.
9. Подключить общий событийный state и связанный text viewer.

## Параллельный запуск субагентов
### Субагент A: backend
- Проверяет источник списка книг.

- Добавляет тесты.

### Субагент B: UI
- Строит dashboard shell.
- Делает карточки и layout.
- Подключает выбор книги.

### Субагент C: smoke

## Жёсткий порядок зависимостей
1. Сначала backend contract.
2. Параллельно UI shell и smoke scaffold.
3. Затем связывание данных, `Book Atlas` и виджетов.
