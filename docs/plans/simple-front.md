# Simple Front

## Цель
Сделать очень простую, надёжную и красивую статическую страницу для визуализации данных из `outputs/` по книгам.

## Контекст
- Данные уже собраны в `outputs/<book>/`.
- Служебные папки `outputs/processed` и `outputs/tables` не являются книгами.
- В репозитории есть старые планы про frontend, webapp и smoke, но для этой задачи нужен другой путь: без web-app, без backend UI, без маршрутизации.
- Нужен именно статический viewer, который быстро открывается и не требует сложной поддержки.

## Диалектический разбор

### Тезис
У нас уже есть богатые выходные артефакты анализа книг: частоты слов, главы, пунктуация, персонажи, связи, тональность, сводные метрики.

### Антитезис
Если пытаться собрать полноценный web-app, задача раздуется: появятся роутер, компоненты, лишний state, сложные экраны, больше точек отказа и больше кода.

### Синтез
Нужно сделать статическую страницу-панель: один селектор книги, несколько хорошо подобранных визуализаций, минимум JS, минимум зависимостей, понятная адаптивная вёрстка.

## Принципы
- Минимум кода.
- Минимум зависимостей.
- Максимум пользы от уже готовых файлов.
- Статическая сборка и публикация на GitHub Pages.
- Никакого backend для рендера визуализаций.
- Никакого SPA и hash-router без необходимости.
- Длинные тексты и таблицы только в скроллируемых панелях.
- Визуализации должны быть разнообразны по типу и смыслу, а не повторять один и тот же график.
- Если файла нет, блок не должен ломать страницу.

## Технический выбор

### Бандлер
Использовать `Vite`.

Почему:
- быстрый production build;
- простой конфиг;
- надёжная упаковка зависимостей;
- удобно работать с `public/` для копирования данных;
- не требует писать свой build pipeline.

### Визуализации
Использовать `ECharts` как основной движок.

Почему:
- bar, line, heatmap, scatter, graph, treemap, pie, area доступны из одной библиотеки;
- декларативный API;
- мало glue-кода;
- подходит для динамической подгрузки JSON/CSV;
- можно добавить `echarts-wordcloud` при необходимости.

### Стили
Использовать готовую базу `Pico.css` и поверх неё маленький локальный `style.css`.

Почему:
- аккуратная типографика без ручной настройки всего подряд;
- простая форма и таблицы;
- хороший базовый баланс для научной презентации;
- свой CSS нужен только для компоновки, карточек и скролл-блоков.

## Анализ данных и смысл визуализаций

Набор визуализаций должен отвечать на разные вопросы:
- что это за книга в целом;
- как устроен её словарь;
- как меняется текст по ходу книги;
- где есть эмоциональные или ритмические сдвиги;
- как устроена персонажная система;
- есть ли необычные участки по пунктуации и редкой лексике.

### 1. Сводные карточки
- Источник: `complexity_metrics.json`.
- Показать: `total_words`, `unique_words`, `hapax_count`, `dis_legomena`, `lexical_density`, `avg_sentence_length`, `total_sentences`, `yules_k`, `honores_r`.
- Зачем: быстрый портрет книги.
- Формат: крупные числа, короткие подписи.

### 2. Топ слов
- Источник: `tokens.csv`.
- Тип: bar chart.
- Показ: top 20 или top 30.
- Зачем: увидеть частотное ядро текста.
- Требование: сортировка по убыванию частоты.

### 3. Rank-frequency / Zipf plot
- Источник: `tokens.csv`.
- Тип: line или scatter в логарифмических осях.
- Зачем: показать характер распределения словаря.
- Особенно полезно для сравнения книг разного размера.

### 4. Ритм книги по главам
- Источник: `chapters_summary.json`.
- Тип: line chart или small multiples.
- Показ: `total_words`, `total_sentences`, `dialog_ratio`.
- Зачем: увидеть архитектуру книги, пики, провалы, диалоговые главы.

### 5. Тональность по главам
- Источник: `sentiment_by_chapter.csv`.
- Тип: line chart.
- Показ: `avg_score`, при необходимости `total_score`.
- Зачем: увидеть эмоциональную дугу текста.

### 6. Пунктуационный профиль
- Источник: `punctuation_counts.csv`.
- Тип: horizontal bar chart.
- Зачем: показать ритм, паузацию и синтаксическую манеру.

### 7. Сеть персонажей
- Источник: `cooccurrence_edges.csv`, при необходимости `characters.csv`.
- Тип: force-directed graph.
- Зачем: раскрыть структуру взаимодействий.
- Требование: фильтрация шумных имен, ограничение по top-N рёбер и top-N узлов.

### 8. Персонажи по главам
- Источник: `character_freq_by_chapter.csv`.
- Тип: heatmap.
- Оси: персонаж x глава.
- Зачем: увидеть, кто доминирует где и как персонажный состав меняется во времени.

### 9. Hapax и редкая лексика
- Источник: `hapax.csv`.
- Тип: compact list или bar chart.
- Зачем: показать уникальный слой лексики.

### 10. Таблица контекстов персонажей
- Источник: `characters.csv`.
- Показ: `name`, `occurrences`, `num_chapters`, `context_sample`.
- Зачем: дать подтверждение и читаемый контекст для имен.

## Дополнительные визуализации
- Scatter `occurrences` vs `num_chapters` из `characters.csv`.
- Heatmap глав по нескольким метрикам: `words`, `sentences`, `dialog_ratio`, `paragraphs`.
- Сравнение книг по сводным метрикам, если добавить общий индекс.

## Дизайн

### Общий стиль
- Светлый фон.
- Один спокойный акцентный цвет.
- Тонкие рамки вместо тяжёлых теней.
- Чистая типографика.
- Много воздуха, но без пустоты.
- Строгая, почти журнальная подача.

### Метафора интерфейса
Не dashboard ради dashboard, а научная презентация книги.

### Компоновка
- Верхняя панель: название, селектор книги, краткий статус загрузки.
- Первый экран: карточки сводных метрик.
- Далее: крупные графики в вертикальном потоке.
- Длинные тексты и таблицы: только внутри скроллируемых панелей.
- Каждая крупная визуализация: отдельный блок с заголовком и коротким пояснением.

### Адаптивность
- Desktop: одна основная колонка с аккуратными секциями.
- Mobile: одна колонка, элементы крупнее, без плотных боковых панелей.
- Для маленьких экранов лучше вертикальная лента, чем сложная сетка.

## Поведение страницы

### Загрузка
1. Открывается список книг.
2. Пользователь выбирает книгу.
3. Загружаются только её данные.
4. Панели рендерятся по мере готовности.

### Кэш
- Данные книги можно держать в памяти на время сессии.
- Повторный выбор той же книги не должен грузить всё заново.

### Ошибки
- Не найден файл: показываем `Нет данных`.
- Ошибка загрузки: компактное сообщение.
- Пустой набор строк: не ломаем макет.

## Что не делать
- Не добавлять backend UI.
- Не делать SPA с роутером ради нескольких секций.
- Не строить отдельный компонентный фреймворк.
- Не тащить тяжёлые зависимости без причины.
- Не повторять одну и ту же диаграмму в разных местах.

## Деплой на GitHub Pages
- Публиковать `site/dist`.
- Перед публикацией копировать все нужные данные в `site/public/data` или аналогичный staging-каталог.
- Страница должна работать статически, без backend.
- На GitHub Pages должен попадать уже собранный результат.

## Проверка

### Ручная
- Селектор книги работает.
- Визуализации меняются при смене книги.
- Большие панели скроллятся.
- На мобильной ширине страница остаётся читаемой.
- Отсутствующие файлы не ломают экран.

### Автоматическая
- `pytest` для сборки и манифеста данных.
- `vite build` для фронтенда.
- Локальный static preview для проверки итогового `dist`.

## Критерии успеха
Страница считается готовой, если она:
- открывает книги из `outputs/`;
- даёт понятный селектор книги;
- показывает разнообразные визуализации;
- выглядит как чистая научная презентация;
- остаётся быстрой и надёжной;
- собирается минимальным кодом;
- деплоится на GitHub Pages без backend.
